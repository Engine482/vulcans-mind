# 05. Intelligence, prompt and RAG tasks — Vulcan’s Mind

## Мета

Зробити Vulcan’s Mind розумнішим без простого переходу на дорожчу модель як першого кроку. Основний приріст якості має прийти від:

1. кращого system/behavior prompt;
2. canonical knowledge documents;
3. curated RAG-бази;
4. synthetic QA / golden answers;
5. нормальної fallback-логіки;
6. можливості легко змінювати модель через env vars.

## P0.1 — Canonical project/profile document

### Завдання

Додати high-priority knowledge document:

```text
vulcan_profile_and_projects.md
```

Цей документ має використовуватися як перше джерело для питань про:

- хто такий Володимир «Вулкан» Моторний;
- над якими проєктами він працює;
- що таке «Люстерко»;
- що таке Vulcan’s Mind;
- які теми бот підтримує;
- які межі має бот.

### Орієнтовна структура документа

```md
# Володимир «Вулкан» Моторний — публічний контекст

## Коротко
...

## Основні публічні проєкти

### «Люстерко»
...

### Vulcan’s Mind
...

### «Іменуй мене»
...

## Дослідницькі інтереси
...

## Що Vulcan’s Mind може пояснювати
...

## Межі відповідей
...
```

### Важливо

Не треба вигадувати біографічні факти. Якщо факт не зафіксований у публічному контексті або затвердженому тексті — не додавати.

## P0.2 — Виправити routing для питань про проєкти

### Завдання

Переконатися, що питання типу:

```text
Над якими проєктами працює Вулкан?
Які проєкти у Володимира?
Що робить Вулкан?
Що таке Люстерко?
Що таке Vulcan’s Mind?
```

не потрапляють у out-of-scope fallback.

### Очікувана поведінка

Такі питання мають діставати контекст із `vulcan_profile_and_projects.md` і давати нормальну відповідь.

## P1.1 — System / behavior prompt rewrite

### Завдання

Переписати prompt так, щоб бот:

- був живішим;
- не звучав як документаційний FAQ;
- не вигадував фактів;
- міг відповідати на загальному рівні, коли питання поруч із темою;
- чітко тримав медичні межі;
- не виводив sources як окремий блок;
- не повторював disclaimer без потреби.

### Draft prompt

```text
Ти Vulcan’s Mind — AI-бот на сайті motornyi.com. Твоя роль — пояснювати публічний контекст, проєкти, досвід і дослідницькі інтереси Володимира «Вулкана» Моторного, а також теми, пов’язані з AI-автоматизацією, RAG/LLM, психометрикою, цифровим скринінгом станів, охороною ментального здоров’я, військовим контекстом і human–AI взаємодією.

Стиль: живий, ясний, розмовний, але не фамільярний. Не звучати як сухий FAQ-бот. Пояснювати складні речі людською мовою. Не роздувати відповідь без потреби.

Фактичність: не вигадуй біографічні факти, партнерства, статуси, медичні claims або результати проєктів. Якщо в базі немає даних — прямо скажи, що надійного контексту недостатньо, і запропонуй корисний напрям розмови.

Медичні межі: ти не є медичним сервісом, не ставиш діагнози, не призначаєш лікування і не даєш персональних медичних схем. Можеш пояснювати загальні принципи, дослідження, скринінг, психометрику, нейромодуляцію та роль AI у підтримці фахівців.

Джерела: використовуй доступний RAG-контекст для формування відповіді, але не виводь окремий великий блок джерел. Якщо потрібно послатися на ідею або дослідження, роби це природно в тексті.

Out-of-scope: якщо питання поза основними темами, не відповідай механічно. Поясни межу коротко і запропонуй 2–3 близькі теми, які можеш розкрити краще.
```

## P1.2 — Curated RAG expansion

### Завдання

Розширити RAG-базу не кількістю сирих PDF, а curated-документами.

### Рівень 1 — власні canonical notes

Додати або оновити:

```text
vulcan_profile_and_projects.md
lusterko_public_project_note.md
vulcans_mind_public_project_note.md
imenuy_mene_public_concept_note.md  # якщо вирішено включити публічно
ai_rag_design_note.md
military_context_digital_tools_note.md
mental_health_boundaries_note.md
```

### Рівень 2 — research abstracts + summaries

Для кожної наукової статті, яку бот використовує як джерело, бажано мати структурований документ:

```md
# Назва статті

## Bibliographic info
...

## Abstract
...

## Key ideas
- ...

## What this source supports
- ...

## What this source does NOT prove
- ...

## Relevance to Vulcan’s Mind / Lusterko
- ...
```

### Рівень 3 — full texts

Повні тексти додавати тільки пізніше, якщо:

- є право їх використовувати;
- налаштований нормальний chunking;
- є metadata;
- retrieval не деградує від шуму;
- є план цитування.

## P1.3 — Synthetic QA / golden answers

### Завдання

Додати набір synthetic QA / golden answers для типових питань.

Це не має бути «джерело фактів». Це має бути behavior/evaluation layer або high-priority examples.

### Приклади питань для golden answers

```text
Що таке Vulcan’s Mind?
Над якими проєктами працює Вулкан?
Що таке «Люстерко»?
Чим «Люстерко» відрізняється від медичного сервісу?
Як AI можна використати в охороні ментального здоров’я?
Що таке RAG?
Чому військовий контекст змінює дизайн цифрових інструментів?
Що таке «Іменуй мене»?
```

### Формат

Можливі варіанти:

```text
golden_answers.yaml
```

або

```text
golden_answers.json
```

Приклад:

```yaml
- question: "Над якими проєктами працює Вулкан?"
  answer: |
    З публічного контексту головні проєкти Володимира «Вулкана» Моторного — це «Люстерко» та Vulcan’s Mind...
  tags:
    - projects
    - profile
    - lusterko
    - vulcans_mind
```

## P1.4 — Retrieval priority

### Завдання

Якщо поточна архітектура дозволяє, додати priority/weight metadata для документів.

### Пріоритети

1. Canonical profile/project docs.
2. Project notes.
3. Curated research summaries.
4. Abstracts.
5. Full texts, якщо будуть.

### Очікуваний результат

Питання про Володимира і проєкти не мають губитися серед академічних джерел.

## P1.5 — Model config via env

### Завдання

Винести model names у environment variables.

### Приклад

```env
VM_DEFAULT_MODEL=...
VM_FALLBACK_MODEL=...
VM_EMBEDDING_MODEL=...
```

### Важливо

Не хардкодити назви моделей у коді. Це дозволить легко перейти на сильнішу mini-модель або додати fallback без рефакторингу.

## P2 — Model routing / fallback

Не реалізовувати зараз, якщо це затягує спринт.

На майбутнє:

- simple questions → default mini model;
- складні research/medical-methodology questions → stronger model;
- API error → graceful fallback;
- unknown topic → semantic fallback, а не hallucination.
